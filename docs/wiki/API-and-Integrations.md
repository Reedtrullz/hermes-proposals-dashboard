# API And Integrations

The API exists primarily for the external worker and compatibility with existing proposal clients. Browser-facing forms use redirecting routes to produce clear post-submit navigation.

## Authentication

Hosted deployments normally set `HERMES_REQUIRE_AUTH=1`. API integrations can supply:

```http
X-Hermes-Key: <HERMES_API_KEY>
```

Local development may use `HERMES_REQUIRE_AUTH=0`.

## Proposal Endpoints

| Method | Route | Purpose |
| --- | --- | --- |
| `GET` | `/api/proposals` | List proposal JSON records. |
| `POST` | `/api/proposals` | Create a proposal and return JSON compatibility output. |
| `GET` | `/api/proposals/{id}` | Read proposal context. |
| `PATCH` | `/api/proposals/{id}/status` | Set proposal status. |
| `POST` | `/api/proposals/{id}/metadata` | Update metadata including project compatibility key. |
| `POST` | `/api/proposals/{id}/comments` | Add a JSON/API comment. |
| `GET` | `/api/proposals/{id}/executor` | Read assigned external executor routing information. |
| `POST` | `/api/proposals/dry-run` | Create a test proposal for a CLI agent route. |

## Browser Routes

| Method | Route | Behavior |
| --- | --- | --- |
| `POST` | `/proposals` | Creates one real proposal and redirects to its detail page. |
| `POST` | `/proposals/{id}/notes` | Stores a rendered-safe note and redirects to detail. |
| `POST` | `/proposals/demo` | Creates or reopens the idempotent guided demo. |
| `POST` | `/proposals/demo/reset` | Deletes only demo-linked records. |
| `POST` | `/proposals/projects` | Creates a project. |
| `POST` | `/proposals/projects/{id}/planning` | Creates a waiting worker recommendation proposal. |

## Other Resource Routes

The application also exposes prefixed operations APIs for agents, goals, workflows, approvals, budgets, and usage records beneath `/api/proposals/...`, with compatible legacy aliases for several existing paths.

## Trigger Files

### Real proposal created

File:

```text
$HERMES_HOME/proposals_trigger
```

Content:

```text
<proposal_id>
```

### Real proposal approved

Content of the same trigger file:

```text
APPROVED:<proposal_id>
```

### Non-Hermes executor selected

File:

```text
$HERMES_HOME/proposals_trigger_executor
```

Example JSON:

```json
{
  "proposal_id": "p_abc123",
  "agent_id": "agent_yyy",
  "executor_type": "codex",
  "executor_label": "Codex CLI"
}
```

No executor metadata file is needed for native Hermes agents. Demo proposals do not write live trigger behavior.

## Status Values

Valid proposal statuses are:

```text
waiting, processing, review, approved, implemented, rejected
```

Integrations should set `processing` only when execution is actually active. For new work generated by the UI or API, `waiting` communicates that the proposal is saved but not known to be executing.

## Integration Guidance

- Preserve trigger file names and formats when implementing a worker.
- Use API reads to obtain context rather than scraping rendered pages.
- Return notes and status updates only after real worker actions.
- Resolve approval decisions through the provided endpoints to keep proposal and approval records synchronized.
- Store secrets in runtime configuration, not proposals or source control.
